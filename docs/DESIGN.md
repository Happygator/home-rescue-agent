## Problem Statement
When a household appliance fails (fridge too warm, washer won't drain, dishwasher won't
drain), a non-expert user has two bad options: pay for a service call they may not need,
or self-diagnose by hunting YouTube/forums and driving the whole process themselves. Most
"broken" appliances are trivial fixes (dirty condenser coil, clogged filter, wrong
setting, tripped breaker). This agent takes a user from "X is broken" to a safe fix, or,
if that fails, to a clean escalation to the right repair contact — with the entire repair
persisted so they can stop and resume later.

Context: solo capstone for the Google/Kaggle 5-Day AI Agents Intensive (Vibe Coding),
open-ended track. Google ADK + Gemini. ~2 weeks, one student. Graded on a working agent +
a Kaggle writeup + a ~3-minute demo video.

## What Makes This Cool
Two fused moments:
- **Camera-magic diagnosis** — point the phone at the model/spec plate and the symptom;
  Gemini reads the exact model number and decodes the error code (for supported categories)
  and names the likely fault.
- **The repair that remembers** — a resumable case file. Stop at 9pm, resume at 8am, full
  continuity, like a contractor managing your job across days.

Differentiation (the chain nobody does end-to-end today): *photo of your specific unit →
exact model + error code → guided multi-session safe loop → close the loop to escalation,
with memory.* iFixit/YouTube, RepairClinic/PartSelect, manufacturer support bots, and
raw Gemini each do a slice; none do the whole chain.

Optional flourish if ahead of schedule: **annotated photo back to the user** (draw the fix
location — "vacuum this coil" — on their own photo). Cheapest high-impact upgrade; carried
over from Approach C.

## Constraints
- ~2 weeks, one student, solo.
- Stack: Google ADK + Gemini (multimodal). `adk web` as the demo UI (no custom frontend).
- Graded deliverables: working agent + writeup + ~3-min video. Non-code deliverables are
  pure human wall-clock — budget for them.
- AI-accelerated dev (Antigravity / Codex / Claude Code) is available. It compresses code,
  NOT perception reliability, content correctness, or demo polish (see Cross-Model).
- Safety is non-negotiable: never advise dangerous gas / electrical / water work.

## Premises (agreed in session)
1. **Narrow + deep** to ~3 appliance categories (fridge, washer, dishwasher), not "any
   appliance." Reliability and a tight story win a capstone; breadth loses it.
2. **Thin curated layer + grounding (REVISED; RESOLVED 2026-06-21).** An error-code→step table + safety rules +
   a few vetted sources, with grounding (iFixit API or Gemini's Google Search grounding)
   doing the heavy lifting. A Day-1 **10-symptom spike** decides how thin the curation can
   be — not a big hand-authored KB unless the spike shows grounding is inaccurate/unsafe.
   **Spike outcome:** Gemini-alone (flash-lite, a conservative lower bound) scored 21/24
   (~88%) first-fix correct with 0 unsafe answers and NO knowledge base. Thin KB confirmed —
   keep only the error-code table + safety rules + ~3-5 targeted corrections. See
   `SPIKE_RESULTS.md`.
3. **Draft/simulated outbound in v1.** Email/call/calendar actions are draft-only or
   simulated with explicit user confirmation. Real sending is a clearly-labeled stretch.
4. **Safety is a demo feature.** The agent refuses dangerous work and escalates to a pro,
   and you demo that refusal on purpose (Day 4 quality/security points).
5. **Memory is the headline.** The persistent, resumable case file is the spine of the
   writeup and video, built early and made visible.

## Approaches Considered
### Approach A: The Tight Loop — CHOSEN
Single ADK agent + tools, `adk web` UI. Photo in → read model + error code (+ validation
layer) → thin-KB/grounding lookup → ranked safe fixes via a scripted decision tree → user
performs checks, results ingested → persist to a database-backed `SessionService`
(resumable case file) → draft escalation email when fixes are exhausted. Safety refusal
demoed. Effort M, Risk Low.

### Approach B: The Crew
Multi-agent: Coordinator routing to Diagnostician + Researcher + Scheduler-Comms, visible
handoffs, shared case-file state. Effort L, Risk Med. Maximum course-coverage and writeup
surface, but the most demo-failure surface for a solo build.

### Approach C: The Instrument
Before/after measurement loop (re-photograph symptom → Gemini diffs images → reports trend)
plus annotated-photo-out, built on A's spine. Effort M/L, Risk Med. Most differentiated and
best video; hero depends on multimodal-diff reliability (de-risked by the Day-1 spike).

## Decision (Chosen Approach)
Session *recommendation* was C-on-A's-backbone; the *decision* is **Approach A (The Tight
Loop)** — the builder prioritized certainty of finishing a clean demo over novelty. Sound for
a solo, graded, 2-week build.

**Demo target (stated once, here):** fridge = committed; washer = stretch (only if the spike
result + schedule allow); dishwasher = descoped from the build (it appears in the problem
framing for context only). "3 categories" is the problem space, not the ship list.

**Optionals are gated, not open.** The annotated-photo-out flourish, the before/after loop
(Approach C), and Cloud Run deployment are all OUT of the baseline plan. Hard rule: do not
start ANY optional until fridge end-to-end + the safety refusal + the resume moment are
recorded on video, and then pick AT MOST ONE. This makes the doc's own anti-scope-sprawl
warning a rule, not a vibe.

## Implementation Details (the load-bearing artifacts)

### Case File Schema (the persisted "memory")
One record per repair case, keyed by `case_id` (+ `user_id`). This is the object the resume UX
rehydrates — build it first.
- `case_id`, `user_id`, `created_at`, `updated_at`
- `appliance_type` (fridge | washer | dishwasher), `brand`, `model_number` (validated),
  `serial_number` (optional)
- `symptom_text` (user's words), `error_code` (nullable)
- `photos[]` — refs to plate photo + symptom photo(s), each with a timestamp
- `steps[]` — ordered `{step_id, instruction, asked_at, user_result, outcome}`, where
  `outcome` ∈ `resolved | not_resolved | unsure | skipped`. Map `user_result` → `outcome` with
  a simple yes/no confirmation prompt ("did that fix it?"), NOT free-text LLM classification —
  it keeps tree transitions deterministic and is the safer 2-week choice. This yes/no glue is
  the operational heart of the loop; build it with the tree.
- `diagnosis` — current best hypothesis + confidence
- `status` — intake | diagnosing | awaiting_user | resolved | escalated
- `escalation` — nullable `{drafted_email, recipient, sent=false}`

### Diagnosis logic — scripted decision tree (one worked branch)
Fridge too warm (fresh-food side), freezer OK:
1. Packed so airflow/vents are blocked? → clear, recheck.
2. Condenser coils dusty? (safe: unplug, vacuum) → clean, recheck.
3. Door gasket sealing? (dollar-bill test) → reseat/replace seal.
4. Temp dial wrong or recently changed? → set correct, wait 24h.
5. None resolved → escalate (draft service request).
Each supported appliance gets one hand-authored tree like this. It is deterministic and "looks
like reasoning" on camera without a combinatorial re-planner. **Ranking key** for suggested
fixes: safety-permitted first → most likely → least user effort.

### Spike rubric (Day 0-1, settles premise #2)
10 symptoms × 3 conditions (Gemini-alone / Gemini+grounding / Gemini+curated). Score each
*first* suggested fix: 2 = correct & safe, 1 = plausible but incomplete, 0 = wrong or unsafe.
Max 20 per condition. Decision branch:
- Grounding ≥ 16/20 and no unsafe answers → ship on grounding + safety rules; no hand KB.
- Grounding < 16/20 or any unsafe answer → author the curated table for exactly the demo
  symptoms/error codes (~10-20 rows for fridge).
If time-pressed, run the spike on fridge symptoms only.

**Plate-read spike (Day 0, ~1 hour, separate from the 10-symptom spike).** The 10-symptom
spike measures *diagnosis* accuracy, not *perception*. Photograph your fridge's model plate
under ~3 lighting conditions and confirm Gemini reads the model number correctly N/M times.
Perception is the demo hook — attach a number to it, don't assume it.

### Unhappy paths (perception is the highest-risk step)
- Blurry/unreadable plate → ask for a closer, well-lit retake (cap 2 retries, then let the
  user type the model number).
- No plate in frame → guide where it usually is (inside door, back, base).
- **Well-formed-but-wrong model number** (passes regex but is a transposition) → cross-check
  the read model against known units in the curated/grounded set; if it doesn't resolve to a
  real unit, re-prompt rather than proceed on a wrong fix.
- Multiple model matches → ask one disambiguating question.
Validation layer = per-brand regex (refine from the real plates you collect on Day 0) + a
membership check against the curated/grounded set. The regex catches malformed reads; the
membership check catches plausible-but-wrong reads.
- Error-code decoding is scoped to the demo unit's codes in the curated table (a Whirlpool
  "E1" ≠ an LG "E1"); an out-of-table code is surfaced verbatim with a "check this code in your
  manual" prompt rather than guessed.

### Resume mechanism (feasibility-spike this on Day 1)
`adk web` is a dev playground; do NOT assume it exposes "resume case by ID" as a user
affordance. Day 1, verify the exact click-path that produces the resume moment: either
(a) `adk web`'s session selector keyed to your `case_id`, or (b) a minimal relaunch with a
`case_id` parameter that rehydrates the Case File from `DatabaseSessionService`. Assume the BASELINE is a small custom resume
entry point (your own `Runner` + `DatabaseSessionService` that recreates the session from
`case_id` and replays state) — `adk web` keys sessions by its own `session_id`, not your
`case_id`, so mapping/rehydrating is the likely path, not the edge case. Budget a half-day for
it (it's a launch parameter, not a UI, so it doesn't break the "no custom frontend" premise).
The headline feature must not rest on an unverified `adk web` affordance.

### Escalation recipient resolution
Recipient = a static per-brand support contact (name/email/phone) stored in the curated table,
with a user-supplied override. No live "find the right place" search in v1 — drafting is a
convenience, not a discovery engine.

### API quota
Confirm a Gemini billing account or sufficient free-tier multimodal quota before Day 5. The
spike (30+ calls), iterative dev, and many demo rehearsals add up; a rate-limit mid-recording
is an avoidable demo failure.

## Open Questions
- **Spike outcome:** how thin can the curated layer be? Decides KB effort. Run Day 1.
- **Knowledge source default is the curated table, not iFixit.** Plan to ship on a small
  hand-curated table covering exactly the demo symptoms + error codes; treat the iFixit API
  (and Gemini Search grounding) as a bonus layer if access lands in time and the spike rewards
  it. The live demo must never depend on a third-party call being up at record time.
- **Which categories for the demo:** fridge is the safe anchor; add washer only if the
  spike result + schedule allow.
- **Resume UX in `adk web`:** how to show two sessions sharing one case ID on video.
- **Test imagery:** stage a small set of real model-plate + symptom photos early; the
  perception moment is the highest-risk part of the demo.

## Success Criteria
- End-to-end demoable on ≥1 appliance (fridge): photo → correct model + error-code read →
  ≥1 correct, safe fix from a guided loop → case persisted → resumed in a fresh session →
  escalation email drafted when fixes are exhausted.
- Safety: the agent visibly refuses a dangerous instruction and escalates to a professional.
- Spike scores ≥ target on first-fix correctness (e.g. ≥16/20) OR a curated table is
  authored to cover the demo cases.
- The 3-min video shows the camera moment, the resume moment, and the safety refusal.
- The writeup names the differentiation chain and cites the grounding source.
