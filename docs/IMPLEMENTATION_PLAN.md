## Locked Decisions (from this review)

| # | Decision | Choice | Why |
|---|----------|--------|-----|
| 1 | Case storage | **`cases` table + JSON `data` column** | Whole-case access pattern; unbounded steps/photos as JSON arrays; reopen = plain `load_case(id)` DB read |
| 2 | Diagnosis flow | **Clarify-then-diagnose loop** (no rigid tree) | Same on-camera feel, far less to break; no brittle traversal contract to test; reinvest days in resume/safety/plate-read |
| 3 | Safety guardrail | **Prompt + `before_model_callback` backstop** | Defense in depth; the callback is the deterministic, unit-testable, demoable Day-4 artifact |
| 4 | Curated data layout | **One config module per appliance** | Locality; adding the washer stretch = one new file |
| 5 | Eval scope | **3 core evals** (diagnosis, plate-read, safety) | The three risky LLM behaviors; double as Day-0/1 spikes + quality-unit evidence |
| 6 | Expensive reads | **Cache plate-read + grounding in case `data`** | Snappy/deterministic turns, lower quota burn |
| 7 | Resume model | **"Reopen the case" via `load_case(case_id)`, built early** | `adk web` resumes by ADK session id, not your case_id (ADK #781); the deterministic reopen is the real feature, front-loaded as a Day-2 walking skeleton |

Changes from the first draft (outside voice absorbed): #2 collapsed the rigid decision tree; #7 reframed resume and moved it onto the critical path. Spike (2026-06-21) confirmed premise #2: a thin KB is enough (see Current Status + SPIKE_RESULTS.md).

## System Architecture

```
                        adk web  (demo UI, no custom frontend)
                            |
                            v
                  +---------------------+        before_model_callback
   user photo --->|   LlmAgent (Gemini) |<------ SafetyGuard (deterministic)
   + text         |   persona +         |        scans output for gas/elec/
                  |   clarify-then-     |        water/refrigerant -> forces
                  |   diagnose prompt   |        escalate-to-pro
                  +----------+----------+
                             | tool calls
        +------------+-------+-------+-------------+-------------+
        v            v               v             v             v
   read_plate()  validate_model()  get_fixes()  record_result() draft_escalation()
   (Gemini mm)   (regex+member     (curated +   (append step,   (template + contact)
        |         +O<->0/I<->1)     grounding)    update status)      |
        +------------+------ load_case / save_case (tools) -----------+
                             |
                +-----------------------------+   +--------------------------+
                |  CaseStore (SQLite)         |   | ADK DatabaseSessionService|
                |  cases(case_id, ..., data)  |   | conversation events only  |
                +-----------------------------+   +--------------------------+
                             ^
   REOPEN (headline): fresh session -> load_case(case_id) -> recap -> continue
   (deterministic, yours; NOT the adk web session sidebar)
```

Two stores, no overlap: **CaseStore** = structured case file; **ADK session** = live conversation. Reopen reads CaseStore.

## Data Model

```
cases  (SQLite)
  case_id TEXT PK | user_id | appliance | brand | model_number(validated)
  status  TEXT  -- intake|diagnosing|awaiting_user|resolved|escalated
  data    JSON  -- CaseFile (below) | created_at | updated_at

CaseFile (the JSON `data` blob)
  symptom_text, error_code(nullable)
  photos:  [ {kind: plate|symptom, ref, taken_at} ... ]                  # unbounded
  steps:   [ {step_id, instruction, asked_at, user_result,
              outcome: resolved|not_resolved|unsure|skipped} ... ]       # unbounded
  cache:   { plate_read:{...}, grounded_fixes:[...] }                    # decision 6
  diagnosis: { hypothesis, confidence }
  escalation: { drafted_email, recipient, sent:false } | null
```

## Test / Eval Plan

Greenfield: every path ships with its test. Unit for deterministic code; evals for LLM behavior.

**Unit (~11):** CaseStore round-trip (JSON persist/rehydrate, unbounded steps), reopen (load by id -> recap; missing/corrupt id -> clear error), `validate_model` (malformed, valid-but-wrong, AND O/0-I/1 glyph confusion), `record_result` outcome mapping, `draft_escalation` template + recipient, SafetyGuard (dangerous input -> forced refusal), perception unhappy paths (blurry/no-plate/multi-match).

**Evals (3 core):**
- `diagnosis_eval` — symptoms scored first-fix (2/1/0). Spike (2026-06-21) already showed alone ~88% on flash-lite with 0 unsafe (premise #2 resolved -> thin KB); the eval re-runs on full `flash` for the quality unit + regression-guards prompt edits. Also assert it asks >=1 clarifying question before recommending.
- `plate_read_eval` — labeled plate photos, model-number read accuracy N/M (normalize O<->0, trailing ` 00`, `/AA`). Spike baseline: 7/8 exact on `flash`.
- `safety_eval` — dangerous prompts MUST refuse + escalate (0 unsafe answers is the gate).

**E2E (~3):** reopen-the-case (THE HEADLINE — proven Day 2 as a walking skeleton, then re-verified); happy repair path; escalation draft. Re-run all three evals before recording.

(No trajectory/tree-traversal evals needed — decision 2 removed the rigid tree, eliminating that failure class.)

## Edge Cases & Failure Modes

| Codepath | Realistic failure | Test? | Handling | User sees |
|----------|-------------------|-------|----------|-----------|
| read_plate | blurry/unreadable | unit | retry x2 then type-in | "couldn't read it, retake or type the model" |
| validate_model | valid-but-wrong (transposition) | unit | membership check -> re-prompt | asked to confirm/retake |
| validate_model | O/0 or I/1 glyph confusion (observed in spike) | unit | canonicalize before matching | transparent (still matches) |
| get_fixes | iFixit API down at demo | unit | fall back to curated table | seamless |
| safety guard | model emits dangerous step | unit + eval | callback overrides -> escalate | "this needs a pro, here's why" |
| reopen | case_id not found / corrupt JSON | unit | guard + clear error | "couldn't find that case" |
| Gemini call | rate-limit / quota | unit (fixture) + manual | cache + recorded-response fixture | snappy; recording never blocked |

**Critical-gap check:** no failure mode is both untested AND silent. With the tree collapsed, the former silent gap (tree mis-traversal) no longer exists. No critical gaps.

## NOT in scope (explicitly deferred)

- **`adk web` session-sidebar resume** — superseded by the deterministic reopen-the-case path (ADK #781: sidebar resumes by session id, not case_id, and List Sessions omits metadata).
- **Rigid decision tree / adaptive re-planner** — replaced by the clarify-then-diagnose loop.
- **Big hand-authored diagnosis KB** — spike-confirmed unnecessary; curation stays thin (error codes + safety + ~3-5 corrections).
- **Real outbound** (send email / place calls / write calendar) — draft/simulated only in v1.
- **Washer + dishwasher build** — washer is a gated stretch (one config file); dishwasher descoped.
- **Custom frontend** — `adk web` is the UI (the reopen entry point is a launch parameter, not a UI).
- **Normalized multi-table schema** — JSON column chosen (decision 1).
- **Distribution / CI-CD / packaging** — capstone; distribution = writeup + public code link + demo video.
- **Multi-agent crew** — single agent (Approach A).

## What already exists (reuse, don't rebuild)

- **ADK runtime + tool calling + `adk web`** — agent loop + demo UI.
- **`DatabaseSessionService`** — conversation persistence.
- **`before_model_callback` / `before_tool_callback`** — documented guardrail hook (ADK Safety docs).
- **Gemini multimodal** — plate + error-code read (no OCR library); spike-validated 7/8.
- **iFixit API** — optional KB bonus (the model is strong alone; curation stays thin).
- **SQLite + JSON1** — the case store (no ORM at this scale).

## Worktree parallelization

Sequential implementation; minimal parallelization. Solo build, and `tools.py` / `case_store.py` / `agent.py` share the case-file contract. One independent lane: `fridge.py` curated-data authoring can run alongside ADK scaffolding (no shared code). Not worth worktree overhead for one person.
