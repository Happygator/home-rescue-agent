This document holds the exploratory thinking behind the design — the approaches that were weighed,
the alternatives rejected, and the cross-model brainstorming. The decisions that **survived** live
in [DESIGN_COMPLETE.md](./DESIGN_COMPLETE.md); this is the record of how we got there, kept separate
so the comprehensive design doc stays focused on what was actually chosen.

## 1. Approaches considered

| Approach | What | Verdict |
|----------|------|---------|
| **A — The Tight Loop** | Single ADK agent + tools; photo → model/code read → thin-KB lookup → ranked safe fixes → persisted case file → draft escalation. Safety refusal demoed. | **CHOSEN.** Effort M, Risk Low. |
| B — The Crew | Multi-agent: Coordinator routing to Diagnostician + Researcher + Scheduler-Comms, visible handoffs, shared case-file state. | Rejected. Effort L, Risk Med. Maximum course-coverage and writeup surface, but the most demo-failure surface for a solo build. |
| C — The Instrument | Before/after measurement loop (re-photograph symptom → Gemini diffs images → reports trend) + annotated-photo-out, built on A's spine. | Rejected as baseline; gated optional only. Effort M/L, Risk Med. Most differentiated and best video, but the hero moment depends on multimodal-diff reliability. |

### Why A won
The in-session *recommendation* was C-on-A's-backbone. The *decision* was **Approach A** — the
builder prioritized certainty of finishing a clean demo over novelty. For a solo, graded, 2-week
build, that is mature judgment (optimize for finishing a clean thing over a flashy thing that might
break), not timidity.

The locked consequences of that decision — the fridge/washer/dishwasher demo target and the
"at most one gated optional" hard rule — are recorded in
[DESIGN_COMPLETE.md §3](./DESIGN_COMPLETE.md#3-the-chosen-approach).

## 2. Gated optional ideas (brainstormed, parked)

These were all explicitly held OUT of the baseline plan and may be picked up — **at most one** —
only after fridge end-to-end + the safety refusal + the resume moment are recorded on video.

- **Annotated-photo-out** — draw the fix location on the user's own photo ("vacuum this coil").
  Cheapest high-impact upgrade; carried over from Approach C.
- **Before/after temporal diagnosis** — have the user re-photograph the same symptom after a fix
  and let Gemini diff the images to report a trend, turning memory into a measurement instrument.
- **Cloud Run / Vertex Agent Engine deployment** — the Day-5 prototype→production story bonus.

## 3. Cross-model perspective (independent cold read)

An independent cold read ran in-session (a fresh-context Claude subagent — the Codex MCP failed to
produce a usable read after two attempts, so this is the documented fallback). Highlights:

- **Coolest unconsidered ideas:** (a) before/after *temporal* diagnosis (now parked, §2);
  (b) annotated-photo-out.
- **The tell:** the builder named "camera-magic" first and used the word "blurry" — in love with
  the perception trick. Guidance: make the camera the hook, the case file the spine.
- **Cut the adaptive re-planner** (a combinatorial tar pit); a simpler loop that *looks* like
  reasoning is ~10% the work and demos identically. (This became the gather-then-fix loop —
  decision #2 in the locked design.)
- **The 50% that already exists:** iFixit API (KB seed), ADK `SessionService` (persistence), Gemini
  Vision (plate read). The real engineering to add is a model-number validation/normalization layer
  so a hallucinated model number gets caught and re-prompted.
- **AI-acceleration verdict — STILL STANDS:** AI compresses typing (ADK boilerplate, schema, glue,
  first-draft prompts, the writeup) but NOT perception reliability, content correctness, or demo
  reps — which were already the long poles. Faster code also *tempts* scope sprawl, the actual
  killer of solo 2-week builds.

## 5. The manufacturer-partnership MCP integration (differentiation direction)

The spike result (Gemini-alone ≈88% on diagnosis, premise #2) is genuinely useful, but it also
exposes a strategic problem: **if a thin prompt + a strong model gets you most of the way, what
stops the product from being "just a ChatGPT chat with a camera"?** This section evaluates the
answer the builder landed on — a future where the app connects to a **manufacturer-hosted (or
service-network-hosted) MCP server** for the specific manual and the *pre-service workflow* a brand
runs before it will dispatch a technician — and how to **mock that interaction in the capstone demo**
so the architecture visibly designs for it.

### 5.1 Why the partnership *is* the moat (not a feature on top of one)

The earlier "one central database of every model → manual → recommended pre-dispatch actions"
evaluation found three sub-problems. Two are tractable (a model catalog is a scraping slog; manuals
are linkable but copyright-risky to host). The third — *what the company recommends before sending a
repairperson* — **has no public source**, because it is the manufacturer's internal call-center
decision tree. That is not a weakness of the partnership idea; it is the entire thesis. A general
chat can only ever **infer** those steps from manuals and forums. An app calling a sanctioned server
returns the **authoritative** workflow — the same one the phone agent runs. That gap is structural,
not something a better prompt closes.

Five things a partnered server gives the product that a general chat structurally cannot:

| Dimension | General LLM chat | Manufacturer-server app |
|-----------|------------------|--------------------------|
| **Authority / liability** | Hallucinates a step; nobody stands behind it | Steps are OEM-sanctioned; the brand owns correctness |
| **Specificity** | Generic per product *line* | Exact model + firmware + active-recall awareness |
| **Closed loop** | Dead-ends at advice | Checks warranty, then **creates the dispatch ticket** |
| **Feedback** | The conversation is thrown away | Step outcomes flow back → improve the workflow |
| **Safety governance** | Best-effort | The deterministic SafetyGuard becomes a *contractual* guarantee |

The product reframes from "AI that guesses your fix" to "**the authenticated front door to
manufacturer service**" — warranty-aware and dispatch-integrated. This is the same chain
[DESIGN_COMPLETE.md §1](./DESIGN_COMPLETE.md#1-product-summary) already claims (photo → exact model →
safe loop → service-ready handoff); the partnership API is what makes the *handoff* end authoritative
instead of best-effort, and is the productized future answer to the "per-company intake forms" gap
[DESIGN_COMPLETE.md §9](./DESIGN_COMPLETE.md#9-diagnosis-flow--curated-data) calls out.

### 5.2 Is the real partnership plausible? Follow the truck-roll cost

The economics are real. A field dispatch ("truck roll") costs whoever pays for it well into the
hundreds of dollars, and a large share are avoidable — a clogged drain, a dirty coil, demo/showroom
mode left on (the exact Samsung `OF OF` case already in the curated table). Call deflection and
first-call resolution are established, multi-billion-dollar cost centers in appliance service.
Anything that resolves issues *before* dispatch is money the partner keeps.

The strategic nuance: **partner with whoever bears the truck-roll cost**, which is not always the
OEM.
- **In-warranty:** the manufacturer eats the dispatch → strong incentive.
- **Extended warranty / protection plans** (Asurion, Assurant, ServicePower-style networks): bear the
  cost and have the *strongest* deflection incentive — often a hungrier, faster first partner than a
  giant OEM.
- **Out-of-warranty:** the consumer pays → the OEM's incentive shifts to brand/CX and parts sales.

Honest hard parts (this is why it stays a *direction*, not a v1 commitment):
- **B2B sales cycle + liability** — "who is liable if your app told them to do step 4 and they got
  hurt" is the gating conversation. The existing hard SafetyGuard is the answer to it.
- **No standard schema** — every OEM's data is shaped differently. **The actual product is the
  normalization layer + a standard MCP server contract** (resource + tool shapes) that partners
  implement; that is the defensible asset.
- **Chicken-and-egg** — OEMs want consumer reach first, consumers want brand coverage first. Wedge in
  with *one* partner (likely a warranty network), prove deflection numbers, expand.

So: hard to *land*, but the value is real and quantifiable, and the integration surface is small.

### 5.3 The demo mock — the highest-leverage capstone move

Mocking the partner server as a **standalone MCP server the agent connects to via ADK's
`MCPToolset`** is the single best demo move, for three reasons:

1. **It demonstrates exactly what the Kaggle agents track judges** — agent *tool use against an
   external system over a standard protocol (MCP)*, not LLM-wrapping.
2. **It makes the architecture legible** — the demo visibly shows a seam where a real OEM plugs in.
   That is the "this is a product, not a prompt" signal.
3. **It bolts onto the existing seam with almost no churn.** `lookup_fixes` → `get_fixes`
   ([grounding.py](../home_rescue/grounding.py)) already isolates the data lookup; registering
   the mock as an `MCPToolset` so its tools sit behind that lookup moves "hardcoded KB" → "partner
   integration" without touching the agent loop (ADK speaks MCP natively, so it is configuration, not
   a hand-written client). A graceful fallback to the thin curated layer keeps it consistent with the
   existing unhappy-path rule (*lookup/grounding down → curated table*,
   [DESIGN_COMPLETE.md §11](./DESIGN_COMPLETE.md#11-perception--capture-unhappy-paths)) so a demo never
   hard-fails if the mock isn't up.

Mock surface (a standalone **OEM MCP server**, canned fixtures keyed off the models already in
`SUPPORTED_MODELS`) — the manual is authoritative context, the workflow and dispatch are callable
tools:

```
get_manual(model)                            → product line, manual_url, warranty status, active recalls
get_pre_service_workflow(model, symptom,     → ordered OEM-sanctioned steps; each with a safety flag,
  error_code)                                  expected outcome, and a terminal `dispatch_recommended` node
create_service_request(...)                  → returns a real ticket id (replaces the escalation-email
                                               draft with a structured, warranty-aware handoff)
```

The manual is exposed as a `get_manual` tool (and optionally an MCP `manual://{model}` resource),
because ADK's `MCPToolset` surfaces *tools* to the model, not resources — a small but real
implementation point for anyone wiring this up.

Demo narrative: *"In production this server is hosted by the manufacturer or the warranty network;
here it's a mock speaking the same protocol. The agent embeds no repair knowledge — it authenticates,
pulls the model's sanctioned workflow, runs the safe loop, and books the truck only when the workflow
says to."* The `get_pre_service_workflow` tool is the authoritative version of today's
`SYMPTOM_FIXES`; the `create_service_request` tool is the authoritative version of `draft_escalation`
— i.e., the service-ready inspection packet (§9) handed to a partner that can actually schedule.

**Effort:** ~1 day — one small MCP server (3 tools + fixtures for the current models), an
`MCPToolset` wired into the agent, and the curated-layer fallback.

### 5.4 Relationship to the locked plan

The **real partnership is post-capstone product direction**, not a v1 deliverable — v1 ships the thin
curated layer that the spike validated. The **mock** is a candidate for the *single* gated optional
and is bound by the same hard rule as every other optional
([DESIGN_COMPLETE.md §3](./DESIGN_COMPLETE.md#3-the-chosen-approach)): nothing optional starts until
the core (fridge E2E + safety refusal + resume + inspection-packet escalation) is recorded, and then
**at most one** is picked — the server mock competes with annotated-photo-out and the before/after
loop (§2) for that one slot. Its advantage over those is that it directly rebuts the "why not just
ChatGPT" critique, which is the strongest thing the writeup can do. The locked, focused version of
this direction lives in
[DESIGN_COMPLETE.md §16](./DESIGN_COMPLETE.md#16-forward-direction-manufacturer-partnership-mcp-server).

## 6. Client/server split — thin client, thick server (recorded 2026-06-24)

**Decision:** the mobile app ([architecture.svg](./assets/architecture.svg)) is a **thin client**;
all agent orchestration — the
model calls, tool execution, and the tool/MCP client that reaches the manufacturer server — runs in
the **FastAPI backend**. The phone captures photos/video, drives the chat UI, and shares the packet;
it never runs the agent loop or holds a connection to a manufacturer server.

### 6.1 Why — the binding constraint is ADK, not the model

A stateless model does *not* require a backend. Because Gemini keeps no session between calls, any
client that can hand it the full context + tool list per turn could drive it — so a backendless
"thick client" (the phone calls Gemini directly, is its own tool/MCP client, persists locally) is a
coherent design. Statelessness is in fact what *enables* a thin client; it argues for the phone, not
against it. The intuition that "stateless model ⇒ no backend needed" is correct as far as the *model*
goes.

What actually pins the orchestrator to a server is the framework: **ADK is server-side Python.** The
agent loop — and the tool/MCP client co-located with it (`McpToolset`, `lookup_fixes`/`get_fixes`) —
runs in a Python process, and there is no ADK-on-device for a Dart/Flutter app. So the agent, and
everything that lives inside its loop, runs server-side. "MCP client on the backend" is a consequence
of *choosing ADK*, not of the model being remote or stateless.

### 6.2 Alternative weighed — thick client / no backend

| | Thin client + thick server (**CHOSEN**) | Thick client / no backend |
|---|---|---|
| Orchestration | ADK in FastAPI | Rewritten in Dart on-device (no ADK) |
| Gemini access | Server holds key/credentials | Direct via Firebase AI Logic + App Check |
| Manufacturer server | Any transport (stdio or remote HTTP) | Remote HTTP only — no local process host for stdio |
| Safety guard | Server-authoritative, tamper-resistant | On-device, bypassable by a modified client |
| Persistence | Server-side `CaseStore` (cross-device, backed up) | Device-local SQLite; no cross-device, no backup |
| Secrets / observability | Centralized (OpenTelemetry, Cloud Logging) | On the device |

Rejected because it discards ADK (the entire build), makes the deterministic SafetyGuard bypassable
— unacceptable for a safety-critical app, and directly at odds with §5.1's plan to harden that guard
into a *contractual* guarantee — and strands all state on one device, when a server-side `CaseStore`
is what gives the user cross-device continuity, backup, and room for any future web client over one
agent. The thick-client design is the right call only for a privacy-first, single-device product,
which this is not.

### 6.3 Grading the reasons (so the decision stays revisitable)

- **Hard:** ADK is server-side — binding for as long as the agent is ADK-based.
- **Strong, project-specific:** tamper-resistant safety enforcement; a server-side shared store for
  cross-device continuity and backup.
- **Mitigable:** key secrecy (Firebase AI Logic solves it client-side) and centralized observability
  — real benefits, not blockers on their own.

If orchestration were ever re-platformed off ADK (e.g. a Dart on-device agent), only the *hard*
reason disappears; the safety and shared-store reasons would still favor keeping the client thin.
